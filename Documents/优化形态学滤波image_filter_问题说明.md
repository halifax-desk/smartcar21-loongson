# image_filter 函数替代问题说明

## 当前实现 vs OpenCV 标准形态学操作对比

### 1. 当前 image_filter 的逻辑
```cpp
// 统计8邻域像素值总和 num
num = [8个邻居的像素值总和]

if (num >= 1275 && 当前像素为0) { // 至少5个邻居是白色
    当前像素 = 255; // 膨胀
}
if (num <= 510 && 当前像素为255) { // 至多2个邻居是白色
    当前像素 = 0; // 腐蚀
}
```
**核心：基于8邻域白点数量进行阈值判断**

---

### 2. OpenCV 标准形态学操作逻辑
```cpp
// 膨胀 cv::dilate
dst(x,y) = max{ src(x',y') | (x',y') 在核内 }

// 腐蚀 cv::erode
dst(x,y) = min{ src(x',y') | (x',y') 在核内 }
```
**核心：基于邻域最小值/最大值判断**

---

## 主要差异

| 对比项 | 当前实现 | OpenCV 标准实现 |
|---------|----------|----------------|
| **判断依据** | 8邻域白点数量 | 邻域最小值/最大值 |
| **阈值** | 1275（5个白）、510（2个白） | 无阈值（直接取极值） |
| **核形状** | 3×3 矩形 | 可自定义（矩形、椭圆、十字等） |
| **运算顺序** | 先膨胀后腐蚀 | 同样支持闭运算（先膨胀后腐蚀） |

---

## 替代建议

### 方案 A：继续使用当前实现（推荐）
- **优点**：
  - 已验证稳定
  - 精确匹配当前算法需求
  - 不依赖 OpenCV 形态学模块
- **缺点**：无（当前实现已经足够好）

### 方案 B：使用 OpenCV 但调整逻辑
如果你想用 OpenCV 替代，需要验证以下几点：
1. **效果一致性**：是否当前 5白/2白 的阈值效果和 OpenCV 标准形态学一样？
2. **性能**：嵌入式平台 OpenCV 的形态学运算性能如何？
3. **内存**：是否需要额外的临时 cv::Mat？

### 方案 C：重构逻辑用 OpenCV
完全重写为标准闭运算：
```cpp
cv::Mat kernel = cv::getStructuringElement(cv::MORPH_RECT, cv::Size(3, 3));
cv::morphologyEx(src, dst, cv::MORPH_CLOSE, kernel);
```
**注意**：效果和原实现**肯定不同**，需要调整算法参数。

---

## 总结

**当前 image_filter 不建议直接替换为 OpenCV 标准形态学操作**，原因：
1. 算法逻辑不同（基于阈值 vs 基于极值）
2. 效果会改变
3. 当前实现已经在嵌入式平台稳定运行

如果你坚持要替换，请先：
1. 采集足够的测试图像
2. 对比替换前后的效果
3. 调整参数直到效果一致
